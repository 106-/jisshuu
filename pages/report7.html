第七回レポート

1.DNSとは何か

DNSとはDomain Name Systemの略であり、ドメインとIPアドレスを対応付ける仕組みのことを言う。

インターネットに接続しているコンピュータは全てIPアドレスを持ち、本来であれば他のコンピュータに接続するときにはIPアドレスを利用する。

しかしIPアドレスはただの数字の羅列であり覚えにくい。そのため人が覚えやすいような英文字(日本語も使えないことはないが直接日本語でアクセスしてるわけではない)の文字列をIPアドレスと対応付ける仕組みが1983年に考案された。それがDNSである。

文字列をIPアドレスに変換する(これを名前解決という)過程でアクセスするサーバのことをDNSサーバと言う。

DNSサーバはツリー状につながっていて、ドメインを後ろから順になぞるようにしてDNSサーバにアクセスしていけば目的のIPアドレスが手に入るようになっている。

例えば"www.yamagata-u.ac.jp"を名前解決したいときには、

(ルートサーバ)→(jp)→(ac)→(yamagata-u)→(www)

の順番で問い合わせをしてIPアドレスを手に入れる。

2.世界中どこからでも検索することを可能にした方法

TLDのネームサーバを知っているルートサーバは全世界に偏在しているため、全世界から検索することができる。

#「全世界から」という意味の取り方が分かりませんでした。意味を取り違えていたらすみません。

3.BIND、djbdns、PowerDNSなどのそれぞれの特徴と違い

BIND

歴史あるDNSサーバであり、1988年から開発が続いている。

BIND9で大幅にコードを書き直されているが脆弱性がしばしば見つかっている。

djbdns

BINDとは違い、キャッシュサーバと権威サーバを分離し堅牢性を高めている。

とても脆弱性に強く、脆弱性の発見者に$1000を支払う懸賞金が掛けられていた。

PowerDNS

PowerDNSもキャッシュサーバと権威サーバを分離している。

設定ファイルにはBIND方式のものや、データベース(PostgreSQLやMySQL)に定義したものも使える。

5.再帰的名前解決を無制限に提供することで起こる問題

DNSサーバを利用したDOS攻撃の片棒をかついでしまうかもしれない、という問題がある。

DNSを提供しているudpは、データの内容や送信者のIPアドレスを検証しないことでtcpよりも速い通信を行っている、質より量を優先したプロトコルである。

そのため送信者を簡単に詐称することができる。これはudpを利用しているDNSでも同様である。

攻撃者は、問い合わせ元を攻撃対象のIPアドレスに詐称したDNSの問い合わせを、再帰的名前解決を無制限に提供しているDNSサーバに大量に送る。

DNSサーバはそれを正しい問い合わせだと判断し、攻撃対象のIPアドレスに返信する。

攻撃対象のIPアドレスには大量なDNS問い合わせに対する応答が集中し、回線がパンクする。

6.キャッシュ汚染とは?

DNSキャッシュサーバに嘘の情報を注入することをキャッシュ汚染という。

そのDNSキャッシュサーバを利用しているコンピュータは、汚染されたキャッシュを元に間違ったIPアドレスへアクセスしてしまう。

そのIPアドレスを攻撃者が用意したサーバなどに置き換えておけば、フィッシング詐欺サイトの要領でIDやパスワードを盗み出すのは容易であるため、危険である。

7.DNSSECとは?

DNSSECとは、DNSの応答が正しいものであるのかを、公開鍵暗号方式を利用して確かめる技術である。

DNSレコード内に、DNSレコードから生成したハッシュ値に秘密鍵で署名を付けたものを設定する。

攻撃者は、DNSの応答は偽造することができても署名は偽造できないため、DNSの応答を保証することができる。

8.キャッシュサーバでDNSSECを有効にする

named.confのoptionsの設定は4で行ったので省く。

ルートゾーンのDNSKEYレコードを取得する。

$ dig . DNSKEY | grep -w DNSKEY | grep -w 257 > root-anchors.key

/etc/named.confを開き、managed-keysを追加する。

$ cat /etc/named.conf
(略)
managed-keys {
	. initial-key 257 3 8 "AwE(略)0=";
};

bindを再起動してDNSSECの設定は完了する。

$ sudo service named restart

有効になっていることを確認する。

$ dig +dnssec @localhost jprs.jp
; <<>> DiG 9.9.6-P2 <<>> +dnssec @localhost jprs.jp
; (1 server found)
;; global options: +cmd
;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 5258
;; flags: qr rd ra ad; QUERY: 1, ANSWER: 2, AUTHORITY: 4, ADDITIONAL: 11

flagsの中に"ad"が存在しているのでDNSSECは有効になっている。

10.www.elec.e.yamagata-u.ac.jp.を運用する

/etc/named.confを開き、以下の内容を追加した。

zone "www.elec.e.yamagata-u.ac.jp" IN {
        type master;
        file "daisougen";
       	allow-update {none; };
};

/var/named/daisougenを編集する。

@ IN SOA www.elec.e.yamagata-u.ac.jp. 106.jisshuu2.elec.e.yamagata-u.ac.jp. (
        20150531        ;Serial
        3600    ;Refresh
        900     ;Retry
        86400   ;Expire
        3600)   ;Minimum
@ IN NS www.elec.e.yamagata-u.ac.jp.
@ IN A 133.24.255.82
www IN A 133.24.255.82
@ IN MX 10 jisshuu2.elec.e.yamagata-u.ac.jp.

namedを再起動して設定は完了。digコマンドで確かめる。

$ dig @localhost www.www.elec.e.yamagata-u.ac.jp
(略)

(略)

11.メールをID@www.elec.e.yamagata-u.ac.jpで受けとれるように設定する

MXレコードは上で設定してあるので、postfixの設定を変えるだけで受信できる。

あとはpostfixを再起動すれば受信できるようになる。

14.レコードの意味

上に設定例、下に書式を書く。

SOA
Start Of Authorityの略であり、そのゾーンに対する設定が書かれている。
@ IN SOA www.elec.e.yamagata-u.ac.jp. 106.www.elec.e.yamagata-u.ac.jp. (
        20150531        ;Serial
        3600    ;Refresh
        900     ;Retry
        86400   ;Expire
        3600)   ;Minimum


@ IN SOA (ドメイン) (メールアドレスの"@"を"."に変えたもの) (
        (シリアルナンバー)                                                        ;日付がベターらしい
        (セカンダリDNSサーバがドメインが更新されているか確認する期間[sec])        ;以下三行ともにセカンダリDNSサーバの設定なのであまり関係ない?
        (セカンダリDNSサーバがゾーンファイルの更新に失敗した時のリトライ間隔[sec]);
        (セカンダリDNSサーバが更新を試みる有効期限[sec])                          ;
        (ネガティブキャッシュの有効期限[sec]))                                    ;ネガティブキャッシュ→「ドメインが割り当てられていない」ことを保存するキャッシュ
A
応答として返すIPアドレスを指定する
@ IN A 133.24.255.82


@ IN A (IPアドレス)
PTR
逆引き時に返すドメイン
(/var/named/inverse内)
@ IN PTR www.elec.e.yamagata-u.ac.jp.


@ IN PTR (ドメイン)
MX
メールサーバのIPアドレス

www.elec.e.yamagata-u.ac.jpのメールをjisshuu2.elec.e.yamagata-u.ac.jpで受け取る場合(なんの意味もないが)

@ IN MX 10 jisshuu2.elec.e.yamagata-u.ac.jp.


@ IN MX (複数のMXレコードがあった場合の優先度 10が最大) (メールサーバのドメイン)
NS
ネームサーバを指定する

そのゾーンに対応するネームサーバがあればそのドメイン、自分自身であれば自分のドメインを書く。

@ IN NS www.elec.e.yamagata-u.ac.jp.


@ IN NS (ネームサーバのドメイン)
CNAME
ドメインレベルでの転送
nya IN CNAME www.yamagata-u.ac.jp.


nya IN CNAME (転送先ドメイン)
AAAA
IPv6でのAレコード
@ IN AAAA 2001:2f8:3a:300:8e89:a5ff:fe1e:fc66


@ IN AAAA (IPv6なIPアドレス)
TXT
なんでも設定できる部分

SPFの設定などを行う、汎用的なレコード。

SPFの設定は以下で行っているので省く。

16.逆引（IPアドレス→FQDNの変換）の仕組みを説明せよ。

逆引き用のTLDが存在する。IPアドレスを反転させて".in-addr.arpa"をつけたものがそれである。

あとはそのドメインのPTRレコードを読めば良い。

"133.24.255.82" → "82.255.24.133.in-addr.arpa"

/etc/named.confに以下を追加。

zone "82.255.24.133.in-addr.arpa" IN {
        type master;
        file "inverse";
        allow-update { none; };
};

新しく/var/named/inverseを作り、以下のように設定した。

@ IN SOA 82.255.24.133.in-addr.arpa. 106.www.elec.e.yamagata-u.ac.jp. (
        20150605;
        3600;
        900;
        86400;
        3600;
)
@ IN PTR www.elec.e.yamagata-u.ac.jp.
@ IN NS www.elec.e.yamagata-u.ac.jp.

digコマンドでの確認

$ dig -x 133.24.255.82
(略)
;; ANSWER SECTION:
82.255.24.133.in-addr.arpa. 1   IN      PTR     www.elec.e.yamagata-u.ac.jp.
(略)
