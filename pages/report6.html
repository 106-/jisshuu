第六回レポート

1.MTAとは何か。役割と意義を述べよ。

MTAとは"Mail/Message Transfer Agent"の略である。

MTAは、送られてきたメールが自分宛てのメールであれば、MDA(それぞれのユーザ別にメールを分けるプログラム)に引き渡す。

そうでなければ別のMTAへ転送する働きをしている。

2.MTAとMUAとMSAの違いを述べよ。

MUAは"Mail User Agent"の略であり、ユーザが書いたメールをMTAに送信する役割をする。メーラー/メールソフトと呼ばれているものは主にMUAを指す。OutlookやThunderbirdなどがある。

MSAは"Message Submission Agent"の略である。MUAとMTAの間に存在し、不完全なメールの補完、SMTP認証、スパムメール対策などを行っているプログラム。

3.sendmail、qmail、postfixその他幾つものMTAがあるが、それぞれの特徴と違いを述べよ。

sendmail
長い間使われてきている歴史あるMTA

細々とした設定ができるが故に基本的な設定を行うのが難解になったという経歴を持つ。

古いMTAであるため、近年の需要(スパムメール対策など)をプラグインとして吸収している。

qmail
sendmailを顧みて作られたMTA。

メール配送に関わる機能を小分けするなど、脆弱性に強くなるような工夫がされている。

その為セキュリティ面では堅牢である。

しかしプログラムのライセンス方式が特殊であったため、ディストリビューションに標準で搭載される機会を失った。

現在はその問題は改善されているが、すでにpostfixに人気を取られている。

postfix
qmailを顧みて作られたMTA。

設定ファイルの書き方が簡単であり、サーバ管理者に理解しやすくなっている。

qmailを参考にした、機能別にプログラムを分ける工夫がされており、セキュリティが堅牢である上に機能の拡張が容易である。

様々なディストリビューションに標準で搭載されている。

radish
Winsows上で動くSMTP/POP3サーバ

Windows上でGUIで設定できるが、TLSに対応していないなど実際のメールサーバとしては不都合な点がある。

SMTPとPOP3サーバの両方が１つのプログラムで扱えるので、設定が簡単である。

イントラネットでの運用がメインだろうか。

4.postfixを適切に設定せよ。postfixを稼働させ、メールが正しく送受信できることを確かめよ。

/etc/postfix/main.cfを編集する。

変更/追加した部分のみ載せる。

myhostname = jisshuu2.elec.e.yamagata-u.ac.jp	//ホスト名を指定。jisshuu2だけで十分かもしれない
mydomain = jisshuu2.elec.e.yamagata-u.ac.jp
myorigin = $mydomain
inet_interfaces = all	//どのSMTPサーバからも受け取れるように設定している
mydestination = $myhostname, localhost.$mydomain, localhost, $mydomain	//どのメールを自分宛てとして扱うかの設定
home_mailbox = Maildir/	//メールを保存する場所を指定する。最後の「/」が無ければmailコマンドで読み取れる形式で保存される。

5.SMTP認証を利用できるように設定せよ。

以下をmain.cfに追加。

smtpd_sasl_auth_enable = yes
smtpd_sasl_local_domain = $myhostname
smtpd_sasl_application_name = smtpd
smtpd_sasl_security_options = noanonymous
smtpd_recipient_restrictions = permit_sasl_authenticated,permit_mynetworks,reject_unauth_destination
disable_vrfy_command = yes

Linuxのユーザを使ってログインする場合

/usr/lib64/sasl2/smtpd.confを編集する。(無ければ作成)

pwcheck_method: saslauthd
mech_list: plain login

/etc/sysconfig/saslauthdを編集する

MECH=pam

/etc/pam.d/smtpを以下のように編集する

#%PAM-1.0
auth    required        pam_unix.so     nullok
account required        pam_unix.so

最後にpostfixをrestartすれば完了。

メールアカウント用のデータベースを独自に使う場合

ユーザ名とパスワードを設定する。

# saslpasswd2 -u jisshuu2.elec.e.yamagata-u.ac.jp -c 106
Password:(パスワードを入力)
Again (for verification):(もう一度パスワード)
$ sasldblistusers2	//ユーザ情報の確認
106@jisshuu2.elec.e.yamagata-u.ac.jp: userPassword

6.25/tcp, 465/tcp, 587/tcpをどのように使い分けるか。

25/tcpは従来通りの平文での通信に利用する。

465/tcpは"SMTP over SSL"でのやりとり(最初から全てSSLで暗号化するプロトコル)に利用され、

587/tcpは"STARTTLS"でのやりとり(最初は平文で通信し、途中からSSLに切り替えるプロトコル)で利用される。

9.自分宛のメールをサーバに残し、かつ電話機にも転送したいとする。どのように設定すればよいか。

.fowardファイルを利用しメールを転送する。

ホームディレクトリに".foward"という名前のファイルを作成し、送信したいユーザ名やメールアドレスをコンマ区切りで記入する。

なお、自分のユーザ名を記入するときは、先頭に"\"をつけておく。付けないとメールが永遠にループしてしまう。

10.TLSが使用できるようにpostfixを設定せよ。

keyは前回使ったものを利用した。

以下はmain.cfに追加。

#TLS
smtpd_use_tls = yes
smtpd_tls_cert_file = /usr/share/ssl/server.pem	//.crtファイル
smtpd_tls_key_file = /usr/share/ssl/server.key
smtpd_tls_loglevel = 1	//0-4の数字を指定。0で無効。数字が大きくなるほどログの量が増える。
smtpd_tls_session_cache_database = btree:/etc/postfix/smtpd_scache

同ディレクトリ内のmaster.cfを開き、以下の行のコメントアウトを外す。

smtps     inet  n       -       n       -       -       smtpd
  -o smtpd_tls_wrappermode=yes
  -o smtpd_sasl_auth_enable=yes
  -o smtpd_client_restrictions=permit_sasl_authenticated,reject

11.MTAでの主な迷惑メール対策（SPF、Greylisting、アドレス検証、Greet pause、その他）について説明せよ。可能なら実際に作動させてみよ。

SPF
なりすましを防止する

具体的には、DNSのTXTレコードに以下のように設定する。

v=spf1 +ip4:133.24.255.82 -all

この設定は、ここで指定したIPアドレスのSMTPサーバとドメインを紐付けるものである。

このレコードに載っていないIPアドレスから送られてきたメールはドメインを偽っていると判断できる。

Greylisting
相手がプロトコルに則った反応をするのか検証する。

アクセスを試みてきたクライアントに対し、一度"4.."系のエラーを返す。

通常のメーラーであれば数秒後にメールを再送するが、スパムメールの多くは一度の試行だけで再送を行わないため、結果的に弾くことができる。

簡単にスパムメールを弾くことができるが、当然ながら遅延が発生してしまう。

アドレス検証
存在するユーザか確認する

一度送られてきたメールに対し、そのメールの宛先が存在するのか確認する。

存在することが確認されるまでの間、その送り手からのメール送信はブロックされる。

存在するかわからない宛先に、無差別に送信しようとしているようなスパムメールに有効である。

Greet pause
相手の応答がゆとりあるものか確認する

SMTPサーバに接続すると、まず最初に

220 jisshuu2.elec.e.yamagata-u.ac.jp ESMTP unknown

というようなメッセージが返ってくる。

通常のメーラーであればこのメッセージが返ってくるのを待ち、受信したら次のコマンドを送るようになっている。

しかし、スパムメールであればその間を取らず、220コードが返ってくる前提で次のコマンドを送る傾向にある。

これを逆手に取り、220コードを返す前に送られてきたコマンドをエラーとして処理するようにすれば、スパムメールの送信を妨げることができる。
